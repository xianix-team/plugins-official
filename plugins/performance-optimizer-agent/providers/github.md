# Provider: GitHub

Use this provider when `git remote get-url origin` contains `github.com`.

## How this fits with the rest of the plugin

- **Reading / analysis** — uses **git** against your base branch (same as Azure DevOps and other hosts). See orchestrator Step 3 in `agents/orchestrator.md`. No `gh` needed to fetch patches or file lists.
- **GitHub-specific** — `gh` is used only to (a) resolve the PR number when one was not passed in, (b) read PR labels, (c) post the analysis comment, and (d) open the optimization PR in fix-PR mode.

## Prerequisites for posting / PR creation

- **GitHub CLI** (`gh`) installed: https://cli.github.com
- Authenticated: `gh auth login`, or non-interactive `GH_TOKEN` / `GITHUB_TOKEN`

**Token scopes / permissions:**

| Permission | Access | Purpose |
|---|---|---|
| **Contents** | Read | Read repository code, branches, commit history (analysis only) |
| **Metadata** | Read | Resolve repository metadata |
| **Pull requests** | Read & Write | Fetch PR context, post findings, open the optimization PR (fix-PR mode only) |

For classic tokens: `repo` (private repos) or `public_repo` (public only); `read:org` if the repository is under an organization.

The plugin does **not** use the GitHub MCP server.

## Resolving the PR number (for posting only)

If the user passed a PR number, use it.

Otherwise, for the **current branch**:

```bash
gh pr list --head "$(git rev-parse --abbrev-ref HEAD)" --json number --jq '.[0].number'
```

Or:

```bash
gh pr view --json number --jq '.number'
```

Parse `owner` and `repo` when needed (e.g. for `gh api` calls):

```bash
REMOTE=$(git remote get-url origin)
# https://github.com/org/repo.git  →  owner=org  repo=repo
# git@github.com:org/repo.git      →  owner=org  repo=repo
OWNER=$(echo "$REMOTE" | sed 's|https://github.com/||;s|git@github.com:||' | cut -d'/' -f1)
REPO=$(echo  "$REMOTE" | sed 's|https://github.com/||;s|git@github.com:||' | cut -d'/' -f2 | sed 's|\.git$||')
```

## Reading the PR labels (to confirm fix-PR mode)

Fix-PR mode is **opt-in** and triggered either by the `--fix-pr` flag or by the presence of the `ai-dlc/pr/perf-optimize-fix` label on the source PR.

```bash
gh pr view <pr-number> --json labels --jq '.labels[].name'
```

If `ai-dlc/pr/perf-optimize-fix` is **not** present and `--fix-pr` was **not** passed, stop after posting the analysis report — **do not** proceed to open an optimization PR.

## Posting the "analysis in progress" comment

```bash
gh pr comment <pr-number> --body "$(cat <<'EOF'
⚡ **Performance analysis in progress**

I'm running an analysis-first performance review covering latency, CPU, memory, and I/O patterns in this change. The consolidated report will be posted here when complete — this may take a few minutes.

_No code changes will be made during analysis. A separate optimization PR is created only when the `ai-dlc/pr/perf-optimize-fix` tag is applied._
EOF
)"
```

If posting fails, output one warning line and continue.

---

## Posting the final analysis report

The analysis-first phase posts **one** consolidated comment — not a formal GitHub Review — so that analysis and fix-PR concerns remain cleanly separated.

```bash
gh pr comment <pr-number> --body "<full compiled report body per styles/report-template.md>"
```

After posting:

```
Performance analysis posted on PR #<number>: <N> bottlenecks ranked (<High>/<Med>/<Low>) — https://github.com/<owner>/<repo>/pull/<number>#issuecomment-<id>
```

## Opening the optimization PR (fix-PR mode only)

Enter this section **only** when the orchestrator has handed off to the `fix-pr-author` agent in fix-PR mode.

### 1. Push the optimization branch

```bash
git push -u origin "${NEW_BRANCH}"
```

### 2. Create the PR

```bash
gh pr create \
  --base  "${SOURCE_BASE_BRANCH}" \
  --head  "${NEW_BRANCH}" \
  --title "perf: optimizations for PR #${SOURCE_PR_NUMBER}" \
  --body  "$(cat <<EOF
## Summary

This PR contains a focused, low-risk set of performance optimizations derived from the analysis of source PR #${SOURCE_PR_NUMBER}. All changes are scoped to Quick-win findings ranked by the Performance Optimizer Agent — no architectural rewrites are included here.

## Links

- Source PR: #${SOURCE_PR_NUMBER}
- Analysis comment: ${ANALYSIS_REPORT_URL:-"(not available)"}

## Applied optimizations

${APPLIED_TABLE}

## Not applied

${NOT_APPLIED_LIST:-"_None — all selected Quick-wins applied cleanly._"}

## Verification checklist

- [ ] Unit tests pass locally / in CI
- [ ] Integration tests pass locally / in CI
- [ ] Manual smoke test on the affected hot path
- [ ] Before/after measurement captured for at least one High-impact item
- [ ] No behavior change intended — API contracts unchanged

## Expected impact

${EXPECTED_IMPACT_NOTES}

---

_Generated by the Performance Optimizer Agent._
EOF
)"
```

### 3. Link the new PR back to the source PR

Add a follow-up comment on the **source** PR with a link to the optimization PR:

```bash
gh pr comment "${SOURCE_PR_NUMBER}" --body "🔗 Optimization PR opened with focused, low-risk performance fixes: ${NEW_PR_URL}"
```

### 4. Output

```
Optimization PR opened: ${NEW_PR_URL} — targets ${SOURCE_BASE_BRANCH}, linked to source PR #${SOURCE_PR_NUMBER}
```

If `gh pr create` fails (e.g. branch protection, missing scope), emit one error line and stop. Do **not** force-push, do **not** rewrite the source branch.
